## 引言：从 Skill 到 APP

当前主流的 AI Skill 架构，本质上是"提示词驱动的意图描述"——一份 SKILL.md 告诉大模型"应该做什么"，模型每次如何理解、如何执行，取决于当时的推理路径。这在创意型、探索性的工作中表现良好，但在企业级应用中暴露出根本性问题：**不确定性和不稳定性**。

同一个 Skill，接不同的模型，输出可能截然不同；接同一个模型，两次运行的结果也可能存在差异。如果我们认为 Skill 驱动大模型是新一代软件的基础框架和工作模式，那么当前的 Skill 架构显然还不能实现稳定、符合预期以及确定性的输出。

**APP (Agent Pipeline Protocol)** 是对这一问题的回应。APP 这个名称概括了三个核心维度：**Agent**（AI 智能体驱动）、**Pipeline**（通过实践固化的执行Pipeline）、**Protocol**（经过验证的执行规程）。

> 在传统世界里你纯靠代码构建 app，在新世界里你用自然语言与代码混合驱动来构建。APP 还是那个 APP——变的是构建方式。

## 一、核心比喻：AI 的肌肉记忆

理解 APP 最直觉的方式，是把它和人类的**肌肉记忆**做类比。

一个人学习弹钢琴。最初，每按一个键都需要大脑思考——看乐谱、判断音符、控制手指。这时候弹得慢、容易出错、非常消耗注意力。但经过反复练习，常用的指法组合会形成肌肉记忆——手指自动完成动作，大脑几乎不需要参与。此时弹奏又快又稳定，而且释放出大脑的注意力去处理更高层次的事情（情感表达、即兴变奏）。

**Skill 就是大脑模式**：灵活、能应对各种意外情况，但消耗大量资源（token），速度慢，且每次执行路径可能不同。

**APP 的 Pipeline 就是肌肉记忆**：只能处理练习过的固定动作，但执行极快、几乎不消耗"思考"资源、输出高度稳定。

这个比喻揭示了 APP 和 Skill 之间最本质的关系：**APP 不是 Skill 的替代品，而是 Skill 的进化形态。** 一个 APP 在使用中逐渐积累 Pipeline（形成肌肉记忆），从"什么都需要大脑想"进化为"常见动作自动完成，大脑只处理新情况"。

### 为什么这个比喻很重要

它解决了一个架构设计中的核心纠结：**我们到底应该选择静态Pipeline的确定性，还是 LLM 动态编排的灵活性？**

答案是：不用选，全都要。

人类也是如此——遇到练习过的动作走肌肉记忆，遇到新情况大脑接管。这不是二选一，而是两个系统协同工作。APP 的架构正是如此：高频路径固化为 Pipeline，未知路径退化为 Skill 模式的动态编排。

## 二、问题诊断：不确定性的来源

当前 Skill 架构的不确定性来自多个层次的叠加：

* **模型版本差异**：不同版本的模型（甚至同一模型的不同版本迭代）对相同提示的理解和输出可能不同。
* **推理随机性**：同一模型在不同温度参数下，或即使相同参数下的不同推理路径，输出存在随机波动。
* **上下文敏感性**：上下文窗口中的内容变化会影响 attention 分布，导致输出偏移。
* **指令歧义性**：SKILL.md 中的自然语言指令本身存在多种合理解读方式。

这些误差在单次任务中可能可以接受，但在复合型的 agentic 流程中会**指数级放大**——前一步的输出偏差成为下一步的输入偏差，逐步累积。

**核心诊断**：问题的本质不是 LLM 不确定，而是当前架构**没有隔离不确定性**。传统软件也有不确定的组件（网络请求、用户输入、随机数），但软件工程有一整套方法论来隔离和处理这些不确定性：类型系统、契约、幂等性设计、重试机制、测试覆盖。当前的 Skill 把所有层次都押注在 LLM 的单次推理上，没有任何这样的隔离层。

## 三、APP 的设计哲学：按任务本质分配能力

在深入 APP 的技术设计之前，需要澄清一个关键问题：**一个 APP 里的工作，哪些该交给代码，哪些该交给 LLM？**

### 划分标准：任务的本质，而非"谁更强"

回到肌肉记忆的比喻。钢琴家弹一段练过千遍的音阶，手指自动完成，大脑几乎不参与——这是肌肉记忆的领地。但当他即兴演奏一段从未弹过的旋律时，必须全神贯注地思考——这是大脑的领地。划分的依据不是"大脑能不能弹音阶"（当然能），而是**这个动作是固定重复的，还是需要理解和创造的。**

APP 里的工作遵循同样的划分逻辑：

**固定重复的工作 → Pipeline代码（肌肉记忆）**：数据采集、统计计算、格式校验、报告骨架生成、JSON 组装——这些工作的逻辑每次都一样，不需要"理解"，只需要精确执行。交给代码，零误差、零 token 消耗。**核心原则**：凡是Pipeline代码已经能确定性完成的事，LLM 不应该用概率性的方式重做一遍。 函数已经能算合规率，就不该让 LLM "算"；API 已经能拿到数据，就不该让 LLM "猜"。

**需要理解和判断的工作 → LLM（大脑）**：分析一段代码的功能、判断一条 commit 是否符合规范、理解用户模糊的意图、在多条路径中选择最合理的下一步——这些工作的核心是语义理解，无法用 if-else 穷举，天然属于 LLM。

### 不是"LLM 只做编排"

一个常见的误解是把 LLM 的角色简化为"编排者"——只负责决定调用哪个函数，不参与具体工作。这过于狭隘。LLM 在 APP 中承担的远不止编排：

```
✅ 语义分析：分析一段代码的功能是什么              → LLM
✅ 判断决策：这条 commit 是否符合规范？该走哪条路？  → LLM
✅ 内容生成：为报告撰写摘要和分析意见              → LLM
✅ 意图理解：用户说"最近一周"具体是哪个日期范围？    → LLM
❌ 已有功能：代码已经实现的功能，不该让 LLM 重做一遍 → 代码
❌ 精确计算：统计合规率、生成排名表格               → 代码
❌ 结构组装：拼接报告骨架、格式化输出               → 代码
❌ 数据获取：调 API、查数据库、读文件               → 代码
```

**LLM 既是编排者，也是执行者——只不过它执行的是需要"理解"的任务，而非需要"精确"的任务。** 代码反过来也一样：它执行确定性的工作，但不参与需要语义理解的判断。

### 核心原则：按任务本质匹配能力

这意味着 APP 的设计问题不是"LLM 该不该参与执行"，而是**为每项工作找到正确的执行者——需要理解的交给 LLM，需要精确的交给代码——并在两者之间建立明确的契约，确保 LLM 产出的内容经过程序化验证后才进入下游流程。**

## 四、双模架构与工程落地

### 约束强度的光谱

如何确保 LLM 知道应该调用哪些函数、以什么顺序调用？现有实践提供了三种约束强度不同的方案：

**最弱约束（工具描述）**：只给 LLM 一组工具定义和功能描述，LLM 完全自主决定调用顺序。简单任务可行，复杂流程可能漏步骤或搞错顺序。

**中等约束（SKILL.md）**：在 Prompt 里写明步骤流程，LLM 照着做。本质上是用自然语言写的 Pipeline。大部分时候有效，但 LLM 可能偏离指令。

**最强约束（静态 Pipeline）**：用 YAML 或代码定义执行顺序，由确定性程序驱动。最可靠，但只能处理预定义的流程，无法应对意外情况。

APP 的关键洞察是：**不需要在这个光谱上选一个固定位置。** 一个 APP 应该同时具备两种模式：

### Pipeline 模式（肌肉记忆）

对于已知的任务路径，预先编写 Pipeline。这些 Pipeline 是完全确定性的——代码控制流程，LLM 只在特定步骤被调用来做语义分析或内容生成，其输出经过 Schema 验证。就像肌肉记忆不需要大脑参与——动作自动完成，快且稳定。

### 动态编排模式（大脑思考）

当用户的请求不匹配任何预定义的 Pipeline 时，系统退化为 Skill 模式——LLM 自主判断该调用什么工具、以什么顺序执行。灵活但消耗更多 token，输出稳定性下降。

### 路由层：意图识别与模式切换

用户请求进来时，路由层的职责是理解用户意图，并尝试将其匹配到一条已有的 Pipeline。这本质上是一个分类任务（从 N+1 个选项中选一个），一个轻量快速的小模型完全够用。

命中 Pipeline → 进入肌肉记忆模式，由确定性代码驱动快速执行；未命中 → 退化为大脑模式，由 LLM 驱动灵活应对。路由失败的后果是安全的：该命中的没命中，多花 token 但不影响结果；不该命中的命中了，Pipeline 因输入不匹配而失败，退回大脑模式。

### Day 0：直接生成 Pipeline

**大部分 Skill 的主路径在设计时就已经很清楚了。** 你写一个 Skill 来做代码审查，那"拿 diff → 逐文件分析 → 汇总报告"这条路径不需要运行一百次才能发现——写 Skill 的人从第一天就知道。一个典型的 Skill，90% 的调用走的都是这些已知的固定路径，只有 10% 是需要 LLM 灵活应对的意外情况。

既然主路径是已知的，就没有理由等。**Day 0 就应该把已知路径写成 Pipeline。** Pipeline 的代码由 LLM 生成，开发者只需要描述"这条路径做什么"。生成成本是一次性的几百 token——而这条 Pipeline 每次被调用，都省下了 LLM 重新编排和推理的 token。如果这个 APP 会被调用成千上万次，这个 ROI 是无脑划算的。

Day 0 具体做三件事：

**1. 把代码该做的事从 Skill 里拆出来。** 看现有 Skill 里 LLM 在做什么，把其中固定重复的工作（数据获取、计算、格式化、报告骨架）用代码实现。LLM 只保留需要理解和判断的部分。

**2. 给 LLM 的输出加验证。** 为每个 LLM 产出定义 Schema，加结构验证和内容验证，验证不过就带错误信息重试。

**3. 把已知路径固化为 Pipeline。** 将主路径的执行顺序从"LLM 每次自己决定"变成"代码写死流程，LLM 只在特定步骤被调用"。加路由层做意图匹配。

### 运行期：持续演化

Day 0 之后，APP 已经能稳定高效地处理 90% 的请求了。剩下的 10% 退化为 Skill 模式灵活处理。

演化发生在这 10% 里：如果某类"意外"请求反复出现，说明它正在从"新情况"变成"常见操作"——此时追加一条新的 Pipeline。APP 的 Pipeline 越来越多，能自动处理的场景越来越广，就像一个人的肌肉记忆库在不断扩大。

## 五、验证：让 LLM 的输出可靠

代码部分不需要操心——函数算出来就是对的。真正需要验证的，是 LLM 负责的那些环节。

### 两道验证

**结构验证**：检查 LLM 输出的格式是否合规——必填字段是否存在、数据类型是否正确、枚举值是否在范围内。这是纯 JSON/Schema 校验，成本几乎为零，应该对所有 LLM 输出默认启用。

**内容验证**：检查 LLM 输出的值在业务上是否说得通——50 个学生不能统计出 200 条腿，分析 50 条 commit 不能产出 53 条结果，合规率不能超过 100%。本质是用代码写几条业务断言，把 LLM 幻觉或计算错误产生的明显荒谬拦住。

### 验证失败：重试，然后报错

验证不通过时，处理方式很简单：**将错误信息注入提示词，让 LLM 重新生成**。设最大重试次数（通常 2-3 次），重试耗尽则报错，把失败信息透明地返回给调用方。

至于更复杂的异常策略——降级输出、熔断停机、人工审核——这些是具体业务场景的决策，不应该在协议层面强制规定。APP 提供验证机制和重试能力，业务层根据自身需要决定失败后怎么办。

## 六、展望：自我进化的 APP

前面描述的 APP，Pipeline 的积累依赖开发者的观察和手动追加。但 APP 架构天然具备一个更深远的可能性：**运行时自我优化**——APP 自己观察自己的行为模式，自动将高频的推理路径编译为 Pipeline。

这个思路在计算机科学中有一个经典的对应物：**JIT（Just-In-Time）编译**。

JIT 编译器的工作方式是：程序以解释模式运行，运行时记录热点代码路径，当某段代码被反复执行时，自动将其编译为机器码。之后再走到这段代码，直接执行编译后的版本——更快，且结果一致。APP 的自我优化遵循完全相同的逻辑：

1. **记录调用栈**：APP 在 Skill 模式运行时，记录 LLM 每次的推理路径——调用了哪些工具、以什么顺序、传了什么参数
2. **识别热点路径**：定期分析这些调用记录，找出那些"大模型推理了半天，最后执行路径都一样"的模式
3. **自动生成 Pipeline**：将这些热点路径编译为 Pipeline 代码，注册到路由层

此后同样的请求再进来，路由层命中新 Pipeline，直接走肌肉记忆模式——跳过 LLM 编排，更快更省。

这种自我优化机制继承了 JIT 的几个重要特性：**只优化热点**，低频的意外请求继续走 Skill 模式，没有额外成本；**对用户透明**，优化前后结果一致，只是更快更省 token；**可以安全回退**，自动生成的 Pipeline 如果因输入不匹配而失败，系统自动退回 Skill 模式重新处理。

如果这个愿景实现，APP 将成为一种**自我进化的软件形态**：它不仅能通过开发者手动添加 Pipeline 变得更"熟练"，还能在运行中自主观察、自主优化，持续将概率性的推理路径编译为确定性的执行代码。一个 APP 用得越多，就越快、越稳定、越省资源——真正实现了"肌肉记忆"比喻的终极形态。
